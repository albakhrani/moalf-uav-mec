## 2. Section III — System Model & Problem Formulation (p.4–5)

### 2.A Network Model (§III-A)
3-D operational space `A ⊂ ℝ³`. Three sets:
- `D = {d₁,…,d_N}` — IoT devices (task sources, limited compute).
- `U = {u₁,…,u_M}` — heterogeneous UAVs acting as MEC units.
- `E = {e₁,…,e_E}` — fixed ground edge servers (extra compute).

Discrete time horizon `T = {1,2,…,T}`, each slot length `Δt`. No governing equation (definitional).
**Depends-on:** sets the index ranges for every summation in (14)–(26) and all constraints.

### 2.B UAV Model (§III-B)
Per UAV `u_j`, state characteristics:
- Position `p_j(t) = [x_j(t), y_j(t), z_j(t)]ᵀ ∈ ℝ³` — units: meters.
- Velocity `v_j(t) = [v_{x,j}, v_{y,j}, v_{z,j}]ᵀ ∈ ℝ³` — units: m/s.
- Computing capacity `C_j(t)` — units: GHz (cycles/s).
- Energy capacity `E_{j,max}` — units: Wh / Watt-hours.
- Current energy `E_j(t) ∈ [0, E_{j,max}]` — Wh.
- Energy-harvesting rate `η_j(t)` — Watts.

**(1)** Continuous kinematics: `dp_j/dt = v_j`,  ∀j∈M, ∀t∈T.
**(2)** Discrete position update: `p_j(t+1) = p_j(t) + v_j(t)·Δt`,  ∀j∈M, ∀t∈T.
**(3)** Velocity constraint: `‖v_j(t)‖ ≤ v_{max,j}`,  ∀j∈M, ∀t∈T. — units m/s.

- **Units:** position m; velocity m/s; `Δt` s.
- **Assumptions:** point-mass UAV, instantaneous velocity control, Euclidean norm speed cap. No acceleration/turn-radius/altitude-bound dynamics modeled.
- **Depends-on:** (1)–(3) underpin all *UAV Trajectory Optimization* results (Table VII, p.13: path length −38%, energy use, coverage), the MPC formulation (28), and the appendix proof (71)–(72).

### 2.C Task Model (§III-C)
Task generation = **nonhomogeneous Poisson process**, rate `λ_i(t)` per device `d_i`.

**(4)** Probability of `k` tasks arriving in `[t, t+Δt]`:
`P(X_i(t+Δt) − X_i(t) = k) = ( ∫_t^{t+Δt} λ_i(τ)dτ )^k / k! · exp(−∫_t^{t+Δt} λ_i(τ)dτ)`.
Standard Poisson pmf with time-integrated rate `Λ = ∫ λ_i(τ)dτ`.

Each task `T_{i,k}` is a tuple `(L_{i,k}, W_{i,k}, τ_{i,k}, ρ_{i,k}, t_{a,i,k})`:
- `L_{i,k}` — input size — **MB**.
- `W_{i,k}` — required CPU cycles — **Megacycles**.
- `τ_{i,k}` — deadline — **seconds**.
- `ρ_{i,k}` — priority — (dimensionless).
- `t_{a,i,k}` — arrival time — seconds.

**(5)** Time-dependent task-urgency metric:
`U_{i,k}(t) = ρ_{i,k} · τ_{i,k} / (τ_{i,k} − (t − t_{a,i,k}))`.
- **Assumptions:** urgency → ∞ as remaining time → 0; priority scales urgency linearly.
- **Depends-on:** (4) is the workload generator behind every "1890 tasks/UAV processed" claim (p.18 summary) and all task-completion metrics; (5) supports the priority/urgency-aware scheduling narrative (no result explicitly cites (5) numerically — ⚠️ usage of (5) downstream is not shown).

### 2.D Communication Model (§III-D)
Channel = log-distance gain. For device `d_i` ↔ UAV `u_j`:

**(6)** Channel gain: `h_{ij}(t) = β₀ · ( ‖p_j(t) − q_i‖² + H² )^{−α/2} · ξ_{ij}(t)`.
- `β₀` — channel gain at reference distance (dimensionless ratio).
- `q_i` — position of device `d_i`.
- `H` — UAV altitude (appears as a constant additive term; **m**).
- `α` — path-loss exponent (dimensionless).
- `ξ_{ij}(t)` — small-scale fading (per Table IV: **Rician, K = 15 dB**).

**(7)** Achievable rate (Shannon–Hartley): `R_{ij}(t) = B · log₂( 1 + P_i·h_{ij}(t) / (N₀·B) )`.
- `B` — channel bandwidth — Hz.
- `P_i` — device transmit power — W (Table IV: 0.1 W).
- `N₀` — noise power spectral density — W/Hz.

- **Assumptions:** flat per-link bandwidth `B`; additive white noise `N₀B`; altitude `H` constant; LoS-dominated Rician fading.
- **Depends-on:** (6)–(7) drive transmission delay `T_{trans,ij}` in (15), network-reliability metric (64), and all "Network Reliability / link reliability 95–97%" results; appendix (74) re-derives (7).

### 2.E Computation Model (§III-E)
For task `T_{i,k}` offloaded to UAV `u_j`:

**(8)** Execution time: `t_{exec,ij,k} = W_{i,k} / C_j`. — units: s (Megacycles ÷ GHz·... ⚠️ unit consistency requires `W` in cycles and `C_j` in cycles/s; paper lists `W` in Megacycles and `C_j` in GHz, so a 10⁶ scaling is implicit).
**(9)** Execution energy: `E_{exec,ij,k} = κ_j · C_j² · W_{i,k}`.
- `κ_j` — energy coefficient of UAV `u_j` (J·s²/cycle³-type constant; **value not given**).
- Appendix (75)–(76) justify the `C_j²·W` (i.e. `∝ f³` power) form via CMOS dynamic power.

- **Depends-on:** (8) → completion-time/latency results (Table V, VI); (9) → `J_energy` (16), energy-efficiency results (Tables III, VI, VII), and (60) "energy ∝ distance" empirical fit.

### 2.F Energy-Harvesting Model (§III-F)
**(10)** `E_j(t+1) = min{ E_j(t) − E_{cons,j}(t) + η_j(t)·Δt , E_{j,max} }`.
- `E_{cons,j}(t)` — energy consumed by UAV `u_j` in slot `t` — Wh/J.
- `η_j(t)` — harvesting rate — W (Table IV: 5 W).
- **Assumptions:** battery capped at `E_{j,max}`; harvest accrues linearly over `Δt`.
- **Depends-on:** UAV operating-time sustainability, "37.55% energy improvement / burst-mode" claims (p.7 impact analysis, Table VII).

### 2.G Integration / Task-Management Mechanism (§III-G)
**(11)** Task-offloading decision: `x_{ijk}(t) = arg min_x [ w₁·D_{ijk}(t) + w₂·E_{ijk}(t) ]`.
- `D_{ijk}(t)` — processing delay for task `k` from device `i` to UAV `j` — s.
- `E_{ijk}(t)` — corresponding energy — J/Wh.
- `w₁, w₂` — delay/energy trade-off weights. ⚠️ Same symbols `w₁,w₂` reused with different meaning in (14); local values **not given**.

**(12)** Energy management (dual-mode): `E_j(t+1) = E_j(t) − E_{cons,j}(t) + min{ η_j·P_harv(t), E_max }`.
- ⚠️ Differs from (10): here the harvest term is `min{η_j·P_harv(t), E_max}` (a product `η_j·P_harv`), in (10) it is `η_j(t)·Δt`. `P_harv(t)` is undefined elsewhere. Reconciliation needed.

**(13)** Priority-based resource allocation: `f_{ijk}(t) = C_j(t) · ρ_{i,k} / Σ_k ρ_{i,k}`.
- Allocates UAV `j` compute proportional to task priority share. Units: cycles/s.
- **Depends-on:** fairness/QoS-allocation claims, `J_util` (19).

### 2.H Problem Formulation (§III-H)

**Decision variables:**
1. `x_{ijk}(t) ∈ {0,1}` — offload task `k` from `i` to UAV `j` (binary).
2. `f_{ijk}(t) ∈ [0, C_j]` — compute resource allocated.
3. `p_j(t) ∈ ℝ³` — UAV 3-D position.
4. `y_{ijkl}(t) ∈ {0,1}` — migrate task `k` from UAV `j` to computing node `l` (binary).

**(14)** Master objective (weighted sum, 6 terms):
`min J = w₁·J_task + w₂·J_energy + w₃·J_completion + w₄·J_migration + w₅·J_util + w₆·J_coverage`.

Component definitions:
- **(15)** `J_task = Σ_t Σ_i Σ_j Σ_k x_{ijk}(t)·( T_{trans,ij}(t) + T_{exec,ij,k}(t) )` — total task latency.
- **(16)** `J_energy = Σ_t Σ_j [ E_{flight,j}(t) + Σ_i Σ_k x_{ijk}(t)·E_{exec,ij,k}(t) ]` — total energy (flight + compute).
- **(17)** `J_completion = − Σ_t Σ_i Σ_j Σ_k x_{ijk}(t)·I(T_{comp,ijk}(t) ≤ τ_{i,k})` — (negative of) on-time completions. `I(·)` = indicator.
- **(18)** `J_migration = Σ_t Σ_i Σ_j Σ_k Σ_l y_{ijkl}(t)·(δ_l + ε_l)` — migration cost; `δ_l` migration delay, `ε_l` fixed migration cost (⚠️ values/definitions not given).
- **(19)** `J_util = − Σ_t Σ_j ( Σ_i Σ_k x_{ijk}(t)·f_{ijk}(t) ) / C_j` — (negative of) resource utilization.
- **(20)** `J_coverage = − Σ_t Σ_i min{ max{ I(R_{ij}(t) ≥ R_min) , 1 } , 1 }` — (negative of) coverage. ⚠️ The nested `min{max{·,1},1}` collapses to 1 whenever the inner value ≥1; the intended expression is unclear.

**Constraints (21)–(26):**
- **(21)** `Σ_j Σ_k x_{ijk}(t) ≤ 1`, ∀i∈N, ∀k, ∀t — each task offloaded to ≤1 UAV.
- **(22)** `Σ_i Σ_k x_{ijk}(t)·f_{ijk}(t) ≤ C_j`, ∀j∈M, ∀t — UAV compute capacity.
- **(23)** `E_j(t) ≥ E_min`, ∀j∈M, ∀t — minimum energy reserve.
- **(24)** `‖v_j(t)‖ ≤ v_max`, ∀j∈M, ∀t — velocity limit (restates (3)).
- **(25)** `T_{comp,ijk}(t) ≤ τ_{i,k}`, ∀i,∀j,∀k,∀t — deadline compliance.
- **(26)** `Σ_l y_{ijkl}(t) ≤ 1`, ∀i∈N, ∀j∈M, ∀k, ∀l∈Q_i(t), ∀t — each task migrated to ≤1 node.

- **Depends-on (14)–(26):** the entire optimization the framework claims to solve; every Section VI/VII performance table is reported against this objective. The per-term weights `w₁..w₆` are the central tuning knobs and are **never given numerically** (⚠️, see §6).

## 3. Section IV — Proposed Solution: MOALF-UAV-MEC (p.5–9)

### 3.0 Architecture (hierarchical, closed-loop) — §IV-A/B
Four cooperating layers (Fig. 1, p.6):
1. **MORL** — high-level decision-making (offload/migrate/allocate policy).
2. **MPC** — UAV trajectory optimization.
3. **APSO** — dynamic resource allocation.
4. **Lyapunov** — stability/performance guarantees on queues & energy.
Coordinated via **Algorithm 1** (`MOALF-UAV-MEC`, p.6) and an inter-algorithm message protocol (54).

### 3.A MORL — Multiobjective Reinforcement Learning (§IV-E, p.7)
Multiobjective tabular Q-learning (**Algorithm 2**, p.8).

- **State space** `S = [p_j(t), E_j(t), Q_i(t), R_{ij}(t)]`, ∀i∈N, ∀j∈M, t∈T. (UAV positions, energies, queue states, link rates.)
- **Action space** `A = [x_{ijk}(t), f_{ijk}(t)]`, ∀i∈N, ∀k∈Q_i(t), t∈T. (offload + allocation decisions.)
- **Reward** `R(s,a) = −[ w₁·J_task(s,a) + w₂·J_energy(s,a) + w₃·J_completion(s,a) + w₅·J_util(s,a) ]`.
  ⚠️ Reward uses only **4** of the 6 objectives (no `J_migration`, no `J_coverage`) and skips weight `w₄` (jumps `w₃ → w₅`). Inconsistent with (14) and with "MORL Number of Objectives = 3" in Table IV.

**(27)** Q-update: `Q(s,a) ← Q(s,a) + α[ R(s,a) + γ·max_{a'} Q(s',a') − Q(s,a) ]`.
- `α` — learning rate (Table IV: 0.001).
- `γ` — discount factor (Table IV: 0.99).
- ε-greedy exploration: `ε` init 1.0, decay 0.995 (Table IV).

- **Depends-on:** all "adaptive decision-making / 94.50% completion rate / faster convergence" claims (Tables III, X; Fig. 5); complexity (35)–(37).

### 3.B MPC — Model Predictive Control (§IV-F, p.7)
Trajectory optimization over receding horizon `N` (**Algorithm 3**, p.9).

**(28)** `J_MPC = Σ_{τ=t}^{t+N} [ w₁·J_task(τ) + w₂·J_energy(τ) + w₆·J_coverage(τ) ]`.
- ⚠️ Uses objectives `{task, energy, coverage}` with weights `w₁,w₂,w₆` — a third subset, again inconsistent with the MORL reward subset and (14).
- `N` — prediction horizon (**value not given**; Table IV has no `N`).

- **Depends-on:** UAV-trajectory results (Table VII): path length −38%, energy 82%, load balance 96%, coverage 98.2%; convergence/horizon analysis (52).

### 3.C APSO — Adaptive Particle Swarm Optimization (§IV-G, p.7)
Resource-allocation search (**Algorithm 4**, p.9).
- **Particle representation:** `X = {f_{ijk}(t) | ∀i∈N, ∀j∈M, ∀k∈Q_i(t)}`.

**(29)** Fitness: `F(X) = −[ w₁·J_task(X) + w₂·J_energy(X) + w₅·J_util(X) ]`.
- ⚠️ Yet another subset `{task, energy, util}` with `w₁,w₂,w₅`.
**(30)** Velocity update: `v_{id} ← w·v_{id} + c₁·r₁·(pbest_{id} − x_{id}) + c₂·r₂·(gbest_{id} − x_{id})`.
**(31)** Position update: `x_{id} ← x_{id} + v_{id}`.
- `w` inertia, `c₁,c₂` cognitive/social coefficients, `r₁,r₂ ∈ U(0,1)`. **None of `w, c₁, c₂`, swarm size, or iteration count are given** (⚠️, §6).
- "Adaptive" qualifier: paper says PSO params are adjusted online, but the adaptation rule is **not specified**.

- **Depends-on:** resource-utilization results (Table III 92%, Table VI), `J_util` improvements.

### 3.D Algorithm 1 (top-level loop, p.6) — control flow
```
Require: D, U, E, T (horizon), Δt, system model parameters
Ensure:  optimized decisions x_ijk, f_ijk, p_j
1  Initialize MORL agent, MPC controller, APSO optimizer, Lyapunov optimizer
2  Initialize component weights w_task, w_energy, w_util, w_traj  (⚠️ 4 named weights here, not w1..w6)
3  for each time slot t do
4    S_t ← observe_state(D, U, E, t)
5    x_ijk(t), f_ijk^{rl}(t) ← MORL_decision(S_t)
6    p_j(t) ← MPC_optimize(S_t, x_ijk(t), p_j^{init}(t))
7    f_ijk^{pso}(t) ← APSO_optimize(S_t, x_ijk(t), p_j(t), f_ijk^{rl}(t))
8    is_stable, adjustments ← Lyapunov_check(S_t, x_ijk(t), f_ijk(t), p_j(t))
9    if not is_stable then
10     x_ijk(t), f_ijk(t), p_j(t) ← apply_adjustments(adjustments)
11   end if
12   Execute actions and update environment
13   Adapt weights and learn models
14 end for
15 return optimized_decisions
```
⚠️ Step 2 names component weights `{w_task, w_energy, w_util, w_traj}` — a *fourth* weight scheme, not matching (14)'s `w₁..w₆`, the MORL reward, MPC, or APSO subsets. The mapping between these naming schemes is undefined.

## 4. Lyapunov stability / drift derivation (§IV-H, p.8) — step by step

**Goal:** bound queue/energy growth while pursuing the objective, via drift-plus-penalty.

**Step 1 — Lyapunov function (quadratic).**
**(32)** `L(Q(t)) = ½ · Σ_i Q_i(t)²`.
- `Q_i(t)` — backlog/queue of device (or node) `i` at slot `t`. Sum over all queues.
- Assumption: standard nonnegative quadratic Lyapunov candidate; captures aggregate congestion.

**Step 2 — Lyapunov drift (one-slot conditional expectation).**
**(33)** `Δ(t) = E[ L(Q(t+1)) − L(Q(t)) | Q(t) ]`.
- Expected change in `L` over one slot given current queues.

**Step 3 — Drift-plus-penalty.**
**(34)** `Δ(t) + V · E[ P(t) | Q(t) ]`.
- `P(t)` — penalty (the optimization cost being traded off, i.e. objective terms).
- `V > 0` — drift/penalty trade-off weight (**value not given**, ⚠️ §6). Larger `V` favors penalty (objective) over stability.

**Step 4 — Per-slot minimization (algorithmic use).**
Text (p.8): the Lyapunov optimization "minimiz[es] the drift-plus-penalty expression at each time step," yielding control decisions that bound queues while approaching long-term optimality.
⚠️ **The paper does NOT carry out the standard expansion** of (33) into an upper bound of the form `Δ(t) ≤ B + (queue·arrival/service terms)`, nor does it derive an explicit `B` constant or the resulting `[O(1/V), O(V)]` optimality–backlog tradeoff. The "derivation" stops at (32)–(34) + prose. There is no closed-form control rule produced from the drift bound.

**Supporting appendix material (p.19):**
- **(71)** `∫_t^{t+Δt} dp_j(τ) = ∫_t^{t+Δt} v_j(τ)dτ` → **(72)** `p_j(t+Δt) − p_j(t) = v_j(t)·Δt` (constant-velocity over slot) → justifies discrete update (2). *(Trajectory, not Lyapunov.)*
- **(73)** Poisson pmf re-derivation → justifies (4).
- **(74)** Shannon–Hartley `C = B·log₂(1+SNR)`, `SNR = P_i·h_{ij}(t)/(N₀B)` → justifies (7).
- **(75)** `P = κ·C·V²·f`, **(76)** `P ∝ f³`, then `E = P·t ∝ κ·C·f²·W`, `f ∝ C_j` → justifies (9).

- **Depends-on:** the "stability / bounded drift / 7-step recovery / 97.5% stability" claims (p.7 §1, Table VIII). Because the bound is not derived, these stability numbers are **empirical**, not analytically guaranteed by the shown math (⚠️).
